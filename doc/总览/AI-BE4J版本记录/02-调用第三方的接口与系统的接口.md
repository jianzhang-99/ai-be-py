# 调用第三方的接口与系统的接口

## 文档说明

- 文档目标：梳理原版 `ai-be-master` 在处理用户请求时，向外部系统、第三方服务和基础设施发起的调用。
- 统计依据：
  - `ai-be-master/src/main/java/com/h3y/aibe/core`
  - `ai-be-master/src/main/java/com/h3y/aibe/manager/oss`
  - `ai-be-master/src/main/resources/application.yml`
- 本文重点关注“AI-BE 调别人”的接口，不是“别人调 AI-BE”的接口。

## 1. 外部调用总体分层

| 类别 | 代表系统 | 作用 |
|------|----------|------|
| 业务系统接口 | 运吨吨、货找船大数据、地图截图 | 提供航运业务数据 |
| MCP/工具接口 | `weather_mcp`、其他动态 MCP 服务 | 扩展工具能力 |
| AI 能力接口 | DeepSeek、通义千问、腾讯 ASR、阿里 OCR | 对话、识别、总结 |
| 存储与基础设施 | 七牛 OSS、Milvus | 文件与向量能力 |
| 公网页面抓取 | 长江航道局水位页面 | 水位信息来源 |

## 2. 业务系统接口

### 2.1 运吨吨领舵系统 `pilot.api.url`

配置来源：

```yaml
pilot:
  api:
    url: http://ai-test2.yundundun.com:9002
```

调用服务：`PilotApiService`

#### 2.1.1 通用约定

| 项目 | 说明 |
|------|------|
| 基础地址 | `${pilot.api.url}` |
| 协议 | HTTP |
| Content-Type | `application/json` 为主 |
| 认证头 | `phone: 当前用户手机号` |
| User-Agent | `AI-Service/1.0` |

#### 2.1.2 接口清单

| 方法 | 路径 | 说明 | AI-BE 用途 |
|------|------|------|------------|
| `GET` | `/pilot-api/ai/page` | 查询运单分页 | 历史运单查询 |
| `POST` | `/pilot-api/ai/order` | 创建运单 | 运单录入保存 |
| `GET` | `/pilot-api/ai/getByName?shipName=` | 船舶精确查询 | 查船 |
| `GET` | `/pilot-api/ai/searchShipByName?shipName=` | 船舶模糊查询 | 找船补全 |
| `GET` | `/pilot-api/ai/bigDataShip/getByName?shipName=` | 大数据船舶信息 | 船位/状态查询 |
| `GET` | `/pilot-api/ai/pageWithOrderByCurrentUser` | 当前用户在途订单分页 | 在途监控 |
| `GET` | `/pilot-api/ai/getByGoodsInfoName?name=` | 货物信息查询 | 运单补全 |
| `GET` | `/pilot-api/ai/listByPortLikeName?portName=` | 港口/地址模糊查询 | 地址补全 |
| `POST` | `/pilot-api/screenshot/capture` | 网页截图 | 地图卡片快照 |

#### 2.1.3 典型调用示例

##### 新增运单

```http
POST /pilot-api/ai/order
phone: 17327756086
Content-Type: application/json
```

请求体由 `OrderAddReq -> PilotOrderAddReq` 转换而来，核心字段包括：

- 装货/卸货地点
- 装货日期
- 货物名称、数量、单位
- 船舶列表

##### 地图截图

```http
POST /pilot-api/screenshot/capture
phone: 17327756086
Content-Type: application/json
```

请求体示例：

```json
{
  "url": "https://test2-map.yundundun.com/#/inTransit?v=xxx",
  "width": 486,
  "height": 300,
  "fullPage": false
}
```

### 2.2 货找船大数据系统 `bigdata.api.url`

配置来源：

```yaml
bigdata:
  api:
    url: http://119.3.33.3:18888
```

调用服务：`BigDataApiService`

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/ai/searchShipPageData` | 按条件搜索船舶分页数据 |

调用特征：

- Content-Type：`application/json`
- 超时：`15s`
- 用途：货找船、船舶列表推荐

### 2.3 地图 H5 页面

配置来源：

```yaml
ai:
  map:
    base-url: https://test2-map.yundundun.com
```

调用位置：`MapService`

说明：

- AI-BE 本身不直接调用地图业务 API
- 它会先构造地图 H5 页面 URL，再交给运吨吨截图服务生成图片
- 两个主要场景：

| 场景 | 页面路由 | 用途 |
|------|----------|------|
| `mmsiQuery` | `/#/inTransit?v=...` | 指定船舶在途地图 |
| `areaQuery` | `/#/inTransit?v=...` | 指定区域附近船舶地图 |

## 3. MCP 与工具系统接口

### 3.1 MCP 通用调用机制

调用组件：

- `McpClient`
- `McpHttpTransport`
- `McpSseListener`

调用方式：

| 类型 | 方法 | 说明 |
|------|------|------|
| 建连 | SSE | 先连 MCP 服务的 SSE 端点 |
| 列工具 | `tools/list` | 获取工具列表 |
| 调工具 | `tools/call` | 调用具体工具 |

JSON-RPC 请求示例：

```json
{
  "jsonrpc": "2.0",
  "id": "uuid",
  "method": "tools/call",
  "params": {
    "name": "地名查询天气预报",
    "arguments": {
      "area": "武汉",
      "date": "2026-05-09"
    }
  }
}
```

### 3.2 天气 MCP

调用入口：`WeatherToolService`

| 项目 | 说明 |
|------|------|
| 工具编码 | `weather_mcp` |
| 工具名 | `地名查询天气预报` |
| 作用 | 获取指定地区和日期的天气 |
| 后处理 | 再用 LLM 将结构化天气总结成自然语言 |

调用链路：

1. LLM 从用户输入里提取地区和日期
2. 通过 MCP 调天气工具
3. 解析天气结果
4. 再次调用 LLM 生成对用户可读的总结
5. 输出 `TEXT + WEATHER`

### 3.3 其他 MCP 服务

管理端支持动态配置和连接外部 MCP 服务：

| 管理接口 | 说明 |
|----------|------|
| `/admin/mcpToolConfig/test/{id}` | 测试某个 MCP 配置 |
| `/admin/mcpToolConfig/connect` | 动态连接某个 URL |

这意味着 AI-BE 的外部工具边界不是固定死的，而是可以通过配置扩展。

## 4. AI 能力接口

### 4.1 DeepSeek

配置来源：

```yaml
spring:
  ai:
    deepseek:
      api-key: xxx
```

调用组件：`DeepSeekCaller`

| 项目 | 说明 |
|------|------|
| 调用方式 | 通过 Spring AI `DeepSeekChatModel` |
| 默认模型名 | `deepseek-chat` |
| 使用场景 | 文本生成、水位总结、部分对话输出 |
| 返回形式 | 流式 token / 阻塞文本 |

说明：

- 具体 HTTP 细节被 Spring AI SDK 封装
- AI-BE 侧关注的是：
  - prompt 拼装
  - 流式输出
  - 调用日志记录

### 4.2 通义千问 DashScope

配置来源：

```yaml
ai:
  tongyi:
    api-key: xxx
```

调用组件：`TongyiCaller`

| 项目 | 说明 |
|------|------|
| 调用方式 | DashScope SDK |
| 常用模型 | `qwen-turbo` 等 |
| 使用场景 | 主对话、参数抽取、天气总结、关联问题等 |
| 形式 | 阻塞 / 流式 |

### 4.3 腾讯云 ASR

调用组件：`TencentAsrServiceImpl`

| 项目 | 说明 |
|------|------|
| 调用方式 | 腾讯实时语音 SDK / WebSocket |
| 输入 | PCM 音频字节流 |
| 输出 | 识别文本 |
| 使用场景 | `/ai/input/speech`、航线解析、按 OSS 识别音频 |

AI-BE 行为特征：

- 按帧推送音频
- 等待识别完成回调
- 聚合句子级文本作为最终结果

### 4.4 阿里云 ASR

调用组件：`AliyunAsrServiceImpl`

| 项目 | 说明 |
|------|------|
| 调用方式 | DashScope WebSocket |
| 模型 | `fun-asr-realtime` |
| 使用场景 | 备用语音识别实现 |

说明：

- 从工厂路由上看，当前主用的仍是腾讯 ASR
- 阿里 ASR 更像备用实现或实验实现

### 4.5 阿里云 OCR

调用组件：`AliyunOcrServiceImpl`

| 项目 | 说明 |
|------|------|
| 调用方式 | DashScope 多模态 OCR |
| 模型 | `qwen-vl-ocr-latest` |
| 输入形式 | Base64、图片 URL、Data URL |
| 输出 | 识别出的文本 |
| 使用场景 | `/ai/input/image`、附件 OCR |

## 5. 存储与基础设施接口

### 5.1 七牛云 OSS

配置来源：

```yaml
oss:
  qiniu:
    bucket-name: test-public-ydd
    domain: testpublicimage.yundundun.com
```

调用组件：`QiniuOssService`

| 项目 | 说明 |
|------|------|
| 调用方式 | 七牛 Java SDK |
| 操作 | 上传文件 |
| 输出 | OSS URL + 本地库表记录 |
| 使用场景 | 图片、音频、附件上传 |

AI-BE 的处理步骤：

1. 生成日期目录和随机文件名
2. 上传到七牛 bucket
3. 拼接公网 URL
4. 将文件记录写入本地 `oss` 表

### 5.2 Milvus 向量库

配置来源：

```yaml
milvus:
  host: 60.204.236.96
  port: 19530
```

相关组件：

- `MilvusConfig`
- `MilvusCollectionService`
- `KnowledgeVectorService`
- `MemoryVectorStore`

用途：

| 方向 | 说明 |
|------|------|
| 知识库向量检索 | RAG 召回 |
| 用户记忆检索 | Memory 检索 |
| 文档切片入库 | 知识文档向量化 |

说明：

- Milvus 属于基础设施连接，不是传统 REST 接口
- 但它是 `ai-be` 对外依赖里非常核心的一环

## 6. 公网抓取型外部来源

### 6.1 长江航道局水位页面

调用组件：`WaterLevelToolService`

| 项目 | 说明 |
|------|------|
| 入口页面 | `https://www.cjhdj.com.cn/hdfw/zhdcdyb/` |
| 调用方式 | `Jsoup` 抓取 HTML |
| 作用 | 获取最近水位公告与正文 |

调用链路：

1. 抓取列表页
2. 找最近的详情页链接
3. 抓取详情页正文/表格
4. 把原始材料交给 LLM 做总结

说明：

- 这不是标准开放 API
- 因此稳定性、格式一致性、证书兼容性都比正式接口差

## 8. 对 Python 重构的建议

### 8.1 建议优先抽象成独立网关的外部系统

- 运吨吨 `pilot`
- 大数据 `bigdata`
- MCP `weather_mcp / generic mcp`
- OSS `qiniu`
- OCR `dashscope_ocr`
- ASR `tencent_asr`
- LLM `deepseek / tongyi`

### 8.2 建议统一治理的点

- 所有外部系统都统一超时、重试、熔断、日志格式
- 把“业务系统接口”和“模型能力接口”拆成两层 client
- 把现在散落在业务代码里的 URL、模型名、工具名收敛成配置项
- 对网页抓取类依赖单独标注“不稳定来源”
