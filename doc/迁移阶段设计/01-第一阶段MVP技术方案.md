# 第一阶段 MVP 技术方案

## 1. 文档目标

定义 AI-BE-PY 第一阶段的技术落地范围。

这一阶段不追求功能齐全，而追求三件事：

1. 服务可以本地稳定启动
2. 主链路可以流式返回
3. 至少具备部分航运业务能力，证明 Python 重构方向可行

## 2. 第一阶段目标

### 2.1 业务目标

第一阶段交付一个可演示、可联调、可继续扩展的最小版本，支持：

- 健康检查
- SSE 流式对话
- 基础意图识别
- 通用闲聊 `TALK`
- 天气查询 `QUERY_WEATHER`
- 查船 `QUERY_SHIP`
- 运单信息抽取预览 `SAVE_ORDER`

### 2.2 技术目标

- 统一项目入口与配置加载
- 建立 FastAPI + LangGraph 主流程
- 建立 LLM 适配层
- 建立 Tool 注册与调用机制
- 明确状态模型和 SSE 事件协议
- 为第二阶段预留 RAG、Memory、持久化扩展位

## 3. 第一阶段范围

### 3.1 范围内

#### 基础能力

- `FastAPI` 服务启动
- `/health` 健康检查
- `/api/chat/stream` 流式接口
- `/api/chat` 非流式接口
- `.env` 配置加载

#### 编排能力

- LangGraph 主工作流
- 意图识别节点
- 路由节点
- 工具调用节点
- 响应生成节点

#### 场景能力

- `TALK`
- `QUERY_WEATHER`
- `QUERY_SHIP`
- `SAVE_ORDER`

#### 外部依赖

- 至少接入一个 LLM Provider
- 至少接入一个业务工具能力
- 允许通过 Mock 适配器替代暂未稳定的第三方服务

### 3.2 范围外

第一阶段明确不做：

- MySQL 持久化落库
- Redis 会话缓存
- Milvus RAG 检索
- 多层记忆系统
- 完整 MCP 动态配置中心
- 运价、找船列表、水位、知识库管理后台
- 后台任务队列
- 生产级权限体系

## 4. 为什么第一阶段只做这些

这是一个典型的 Agent 重构项目，第一阶段最容易失败的点不是“功能少”，而是“基础编排不稳”。

如果一开始就同时引入：

- LangGraph
- 多模型
- MCP
- Milvus
- Redis
- MySQL
- 一堆业务系统接口

那排错成本会非常高，最后很容易变成“文档看起来很完整，但项目跑不起来”。

所以第一阶段的设计原则是：

- 主链路先闭环
- 外部依赖先最小化
- 业务价值先可演示

## 5. 第一阶段架构裁剪

### 5.1 总体架构

```text
Client
  -> FastAPI Router
  -> ChatService
  -> ChatWorkflow
      -> intent_node
      -> routing_node
      -> tool_node / response_node
  -> SSE Event Stream
```

### 5.2 分层职责

| 层 | 第一阶段职责 | 暂不承担 |
|----|--------------|----------|
| API | 收请求、参数校验、输出 SSE | 复杂鉴权 |
| Service | 组织工作流调用 | 复杂业务聚合 |
| Graph | 场景编排、状态流转 | 多工作流并发调度 |
| Tools | 调业务工具和外部服务 | 动态工具市场 |
| Infra | LLM/配置/HTTP Client | 完整存储体系 |

## 7. 第一阶段核心状态模型

建议第一阶段不要一上来就做很重的状态对象，只保留主链路必要字段。

```python
AgentState = {
    "request_id": str,
    "session_id": str,
    "user_id": str,
    "user_input": str,
    "history": list,
    "intent": dict | None,
    "scene": str | None,
    "tool_name": str | None,
    "tool_result": dict | list | str | None,
    "response_text": str | None,
    "error": str | None,
}
```

### 设计原则

- 第一阶段状态字段不超过 10 个
- 所有节点只写自己负责的字段
- 不在状态里塞 SDK 客户端、数据库连接等运行时对象

## 8. 第一阶段场景设计

### 8.1 TALK

#### 目标

验证最基础的 LLM 对话链路可用。

#### 实现方式

- 意图识别命中 `TALK`
- 直接走 `response_node`
- 调用 LLM 生成文本

#### 价值

- 可以验证 FastAPI、LangGraph、LLM、SSE 四段链路都已通

### 8.2 QUERY_WEATHER

#### 目标

验证“意图识别 -> 工具调用 -> 结果回传”的工具链路。

#### 第一阶段建议

- 第一优先：实现 `weather.py` 工具，直接通过 HTTP 或 MCP 查询天气
- 第二优先：如果第三方依赖未打通，先提供 Mock 天气结果

#### 返回形式

- 先返回文本总结
- 再返回结构化天气卡片或 JSON

### 8.3 QUERY_SHIP

#### 目标

验证航运领域的第一个真实业务场景。

#### 第一阶段建议

- 输入船名
- 调用 `ship.py`
- 返回单船基础信息

#### 第一阶段范围控制

- 先不做地图截图
- 先不做在途监控卡片
- 先只返回“船名、MMSI、船型、载重”等基础信息

这样可以降低对运吨吨截图和地图链路的耦合。

### 8.4 SAVE_ORDER

#### 目标

验证“从自然语言抽结构化数据”的能力。

#### 第一阶段建议

- 输入一段运单话术
- 调用 `order.py`
- 返回结构化预览结果
- 不真正创建运单

#### 范围边界

第一阶段只做：

- 装货地
- 卸货地
- 装货日期
- 货物名称
- 货量
- 单位

不做：

- 真正保存到外部运吨吨系统
- 多船信息补全
- 二次确认工作流

## 9. 第一阶段 Tool 设计

### 9.1 Tool 注册机制

第一阶段推荐做静态注册，不做动态发现。

```python
TOOL_REGISTRY = {
    "query_weather": WeatherTool(),
    "query_ship": ShipTool(),
    "save_order_preview": OrderExtractTool(),
}
```

### 9.2 Tool 接口约定

```python
class BaseTool(Protocol):
    name: str

    async def run(self, user_input: str, state: dict) -> dict:
        ...
```

### 9.3 为什么先静态注册

- 调试成本低
- IDE 友好
- 对第一阶段足够
- 第二阶段再平滑升级为 registry + factory

### 10.1 健康检查

- `GET /health`
- 返回运行状态、版本、时间

### 10.2 非流式对话

- `POST /api/chat`
- 用于调试和简单联调

### 10.3 流式对话

- `POST /api/chat/stream`
- 使用 `text/event-stream`

建议统一事件类型：

- `intent`
- `tool_start`
- `tool_result`
- `response`
- `done`
- `error`

## 11. 第一阶段外部依赖策略

### 11.1 LLM

第一阶段只保留一个主 Provider，建议：

- 默认 `DeepSeek`
- 预留 Provider 抽象

原因：

- 避免一开始就做多模型路由
- 先把调用、超时、异常处理稳定下来

### 11.2 业务外部系统

第一阶段采用“适配器 + 可 Mock”策略：

| 能力 | 第一阶段策略 |
|------|--------------|
| 天气 | 真实接口优先，失败可 Mock |
| 查船 | 真实接口优先，失败可 Mock |
| 运单抽取 | 本地 LLM 生成即可 |

### 11.3 存储

第一阶段全部做成可选依赖：

- 没有 MySQL 也能跑
- 没有 Redis 也能跑
- 没有 Milvus 也能跑

## 12. 第一阶段异常处理

### 12.1 分层原则

| 层 | 异常策略 |
|----|----------|
| Router | 参数校验失败直接返回 4xx |
| Service | 包装业务异常 |
| Graph Node | 写入 `state["error"]` |
| Tool | 统一抛 `ToolExecutionError` |
| LLM Client | 统一抛 `LLMError` |

### 12.2 SSE 错误输出

即使出错，也尽量输出标准事件：

```json
{
  "event": "error",
  "data": {
    "message": "ship tool timeout"
  }
}
```

然后显式发一个 `done` 结束。

## 13. 第一阶段可观测性要求

第一阶段不做复杂监控，但至少要有：

- request_id
- session_id
- 场景识别日志
- tool 调用耗时
- LLM 调用耗时
- 错误日志

建议日志字段：

| 字段 | 说明 |
|------|------|
| `request_id` | 单次请求唯一标识 |
| `session_id` | 会话标识 |

## 14. 第一阶段完成定义

当以下条件全部满足时，第一阶段算完成：

1. `uvicorn backend.main:app --reload` 能成功启动
2. `/health` 返回正常
3. `/api/chat` 能返回非流式结果
4. `/api/chat/stream` 能稳定输出 SSE
5. `TALK` 场景可用
6. `QUERY_WEATHER` 可用
7. `QUERY_SHIP` 或 `SAVE_ORDER` 至少有一个真实业务场景可用
8. 有基础单元测试和至少一条集成测试

## 15. 结论

第一阶段不是“做一个简化版大系统”，而是“做一个能跑、能演示、能继续长出来的最小系统”。

从技术管理角度，这一阶段最重要的不是功能数量，而是建立以下基础共识：

- 目录结构是否合理
- 编排方式是否可持续
- Tool 模式是否顺手
- 状态模型是否够稳
- 外部依赖是否可控

如果这几个问题在第一阶段验证通过，第二阶段扩展业务深度就会顺很多。

## 16. 第一阶段意图识别模型方案

这一章专门回答一个问题：

如果我们准备拿一个 `Qwen` 7B 到 9B 量级模型做 SFT，来提升航运场景下的意图识别能力，第一阶段应该怎么做。

### 16.1 先说结论

对当前项目，推荐采用下面这个简单判断：

- 第一优先：做模型微调
- 第二优先：把实体抽取一起做进去
- 第三优先：后续再接 RAG

也就是说，第一阶段不要一上来就做“微调 + RAG 大杂烩”，而是先把模型训练这条主线跑通。

原因很简单：

- 微调解决的是“模型会不会理解这个行业里的说法”
- RAG 解决的是“模型有没有额外知识可查”

当前项目里的核心问题，如：

- `俞垛79` 是船名还是地名
- `南京港附近` 是区域找船还是普通地名问答
- `俞垛在哪里` 是否承接上一轮对话

本质上都更偏“理解和判断”，所以第一阶段主线应是微调。

### 16.2 用新手容易理解的话解释

#### 微调是什么

可以把微调理解为“教模型学会你这个行业怎么说话、怎么判断”。

例如训练后，模型要学会：

- `查船 俞垛79` 对应 `QUERY_SHIP`
- `帮我查船：南京港附近` 更像 `FIND_SHIP`
- 上一轮已经查过 `俞垛79`，下一轮 `俞垛在哪里` 仍然应该优先理解为查船

#### RAG 是什么

可以把 RAG 理解为“先去知识库翻资料，再把资料喂给模型回答”。

它更适合这类问题：

- 运吨吨怎么操作
- 某个接口字段是什么意思
- 某条业务规则是什么

所以本项目里：

- `意图识别` 更依赖微调
- `文档问答 / 帮助中心 / 规则说明` 更依赖 RAG

### 16.3 第一阶段是否需要 RAG

第一阶段结论是：

- 可以预留 RAG 扩展位
- 但不把 RAG 作为意图识别主方案

推荐的边界是：

- `QUERY_SHIP`、`FIND_SHIP`、`SAVE_ORDER` 这类任务先靠 SFT 模型识别
- `DOC_QA` 或操作说明类问答，后续阶段再接 RAG

这样做的好处是职责清晰：

- 微调模型负责“判定”
- RAG 负责“补知识”

## 17. 第一阶段模型目标定义

### 17.1 不要只做单标签分类

第一阶段不建议把模型目标定义成：

```text
输入一句话 -> 输出一个 intent 标签
```

因为这不能很好解决你当前的真实问题。

更推荐定义成：

```text
输入：当前用户问题 + 最近 1 到 3 轮上下文
输出：intent + slots + need_clarify
```

示例：

```json
{
  "intent": "QUERY_SHIP",
  "slots": {
    "ship_name": "俞垛79",
    "area_name": null
  },
  "need_clarify": false
}
```

### 17.2 为什么要把槽位一起做

因为业务系统最终不只需要一个意图名，还需要参数。

例如：

- `QUERY_SHIP` 需要 `ship_name`
- `FIND_SHIP` 需要 `area_name`、`route_from`、`route_to`
- `SAVE_ORDER` 需要装货地、卸货地、货名、吨位等字段

如果只训练 intent，不训练 slots，后续还得单独补一套抽取逻辑，系统会更绕。

### 17.3 为什么要加 need_clarify

线上最怕的不是“模型不确定”，而是“模型非常自信地判断错了”。

因此建议模型支持输出：

- `need_clarify = true`

适用场景：

- `南京7` 可能是船名，也可能是地名
- `俞垛在哪里` 可能是问上一轮的船，也可能是问地名

如果模型不确定，系统可以追问：

- `你是想查船“南京7”，还是想查询南京港附近的船舶？`

## 18. 第一阶段推荐技术路线

### 18.1 主线方案

第一阶段主线建议：

- 基座模型：`Qwen` 7B 到 9B 量级指令模型
- 训练方式：SFT
- 训练目标：多轮意图识别 + 槽位抽取 + 澄清判断
- 输出形式：结构化 JSON

这条路线的特点是：

- 对新手相对友好
- 数据准备和评估比较直观
- 后续容易演进到更复杂的 Agent 编排

### 18.2 为什么选 SFT

因为第一阶段最重要的是先把“标准回答格式”和“航运语义理解”训出来。

SFT 适合做这件事：

- 输入格式固定
- 输出格式固定
- 训练目标清晰
- 容易做人工抽查

第一阶段先不引入复杂的 RL、DPO、在线反馈闭环，先把基本可用版本做出来。

### 18.3 为什么不一开始就做纯分类器

纯分类器的问题是：

- 很难直接处理多轮上下文
- 很难顺带输出槽位
- 很难自然支持澄清判断

而 SFT 结构化输出模型可以统一做这些事情。

## 19. 模型输入输出设计

### 19.1 输入格式

建议统一成多轮指令格式：

```text
你是航运助手的意图识别模型。
请根据对话历史，识别用户当前意图，并抽取关键槽位。
如果信息不足或存在明显歧义，请输出 need_clarify=true。

[历史对话]
用户：查船 俞垛79
助手：已为您查询到船舶俞垛79

[当前问题]
用户：俞垛在哪里
```

### 19.2 输出格式

建议统一输出 JSON：

```json
{
  "intent": "QUERY_SHIP",
  "slots": {
    "ship_name": "俞垛79",
    "area_name": null
  },
  "need_clarify": false,
  "clarify_question": null
}
```

如果需要澄清：

```json
{
  "intent": null,
  "slots": {
    "ship_name": null,
    "area_name": "南京"
  },
  "need_clarify": true,
  "clarify_question": "你是想查船“南京7”的位置，还是想查询南京港附近的船舶？"
}
```

## 20. 数据方案

### 20.1 训练数据长什么样

第一阶段建议每条样本至少包含：

- `history`
- `query`
- `intent`
- `slots`
- `need_clarify`

可参考结构：

```json
{
  "history": [
    {"role": "user", "content": "查船 俞垛79"},
    {"role": "assistant", "content": "已为您查询到船舶俞垛79"}
  ],
  "query": "俞垛在哪里",
  "label": {
    "intent": "QUERY_SHIP",
    "slots": {
      "ship_name": "俞垛79",
      "area_name": null
    },
    "need_clarify": false
  }
}
```

### 20.2 第一阶段必须覆盖的样本类型

1. 单轮明确意图
   - `查船 俞垛79`
   - `今天南京天气`

2. 船名与地名歧义
   - `南京7在哪里`
   - `俞垛在哪里`

3. 区域找船
   - `南京港附近有没有船`
   - `帮我查船：扬州港周边`

4. 多轮续问
   - 上一轮查船，下一轮只问 `在哪里`
   - 上一轮找船，下一轮只问 `附近还有吗`

5. 运单抽取
   - 自然语言中的装货地、卸货地、日期、货量

### 20.3 新手阶段的数据建议

如果刚开始样本不多，可以先这样做：

- 先整理 300 到 500 条高质量种子样本
- 再扩展到 1000 到 3000 条可训练样本
- 其中专门抽 100 到 300 条做“歧义专项测试集”

对新手来说，数据质量比一开始盲目堆量更重要。

## 21. 训练与部署方案

### 21.1 第一阶段训练方式

推荐路径：

- 用现成 `Qwen` 指令模型做 SFT
- 优先用 LoRA/QLoRA 训练
- 不直接全参数微调

原因：

- 显存压力更小
- 上手更简单
- 试错成本更低

### 21.2 训练产物

第一阶段希望得到两个结果：

1. 一个能输出标准 JSON 的微调模型
2. 一份评估报告，证明在核心样本上确实比当前规则/裸 LLM 更稳

### 21.3 线上调用方式

建议第一阶段先做成独立识别服务，供现有 LangGraph 调用：

```text
ChatWorkflow
  -> intent_node
      -> IntentModelService
      -> 返回 intent + slots + need_clarify
  -> routing_node
```

这样好处是：

- 不会一下子推翻现有工程
- 可以和旧规则并行灰度
- 容易做 A/B 对比

## 22. RAG 在本项目中的正确位置

### 22.1 RAG 适合放在哪

RAG 更适合这些场景：

- `DOC_QA`
- 操作说明
- 帮助文档问答
- 业务规则解释

例如：

- `运吨吨怎么发布运单`
- `这个字段是什么意思`
- `船舶列表接口怎么用`

### 22.2 RAG 不应该承担什么

第一阶段不建议让 RAG 来负责：

- 主意图识别
- 船名/地名歧义主判定
- 多轮续问主判断

因为这些问题主要不是“缺资料”，而是“需要模型学会行业里的理解方式”。

### 22.3 推荐组合方式

推荐后续组合关系：

```text
用户输入
  -> 意图模型
      -> 如果是 DOC_QA
          -> 再走 RAG
      -> 如果是 QUERY_SHIP / FIND_SHIP / SAVE_ORDER
          -> 走业务工具链路
```

这样组合最清晰，也最容易排错。

## 23. 风险点与缓解

### 23.1 常见风险

| 风险 | 表现 | 缓解方式 |
|------|------|----------|
| 数据太少 | 模型泛化差 | 先做高质量种子集，逐步扩充 |
| 标注不一致 | 同类问法学不稳 | 先写清楚标注规范 |
| 输出不稳定 | JSON 解析失败 | 统一输出模板，训练时强约束 |
| 歧义误判 | 船名/地名混淆 | 增加 `need_clarify` 标签和专项样本 |
| 多轮失效 | 脱离历史理解 | 训练样本必须带上下文 |

### 23.2 第一阶段最容易犯的错

最容易犯的错有两个：

1. 一开始就想同时做微调、RAG、复杂 Agent、所有场景
2. 只标 intent，不标 slots 和歧义澄清

第一阶段更推荐的节奏是：

- 先把识别模型训出来
- 再把评估跑通
- 再决定后续要不要加更多能力

## 24. 第一阶段完成定义补充

除了第 14 章的完成定义，针对模型方案，再补充以下验收条件：

1. 能产出一版结构化训练数据
2. 能完成至少一次 SFT 训练
3. 模型能稳定输出合法 JSON
4. 在核心专项样本上优于当前规则方案
5. 能接入现有 `intent_node` 做本地联调
6. `DOC_QA` 仍保持为后续 RAG 预留场景

## 25. 本章结论

如果用一句最简单的话概括这一章：

- 微调是让模型学会“怎么判断”
- RAG 是让模型获得“可检索的知识”

对当前项目，第一阶段正确顺序应该是：

1. 先做 `Qwen` SFT 意图识别模型
2. 同时把 `slots` 和 `need_clarify` 一起纳入训练
3. 再把模型结果接到现有 LangGraph 编排
4. 最后在 `DOC_QA` 等场景里引入 RAG

这条路线最适合新手启动，也最符合当前项目的真实问题结构。
