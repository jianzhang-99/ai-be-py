# 第一阶段接口与验收

## 1. 接口目标

第一阶段接口设计只服务一件事：让前端或测试方能快速验证 AI-BE-PY 已经跑起来，并具备最小可用能力。

## 2. 接口清单

| 接口 | 方法 | 说明 |
|------|------|------|
| `/` | `GET` | 根路径 |
| `/health` | `GET` | 健康检查 |
| `/api/chat` | `POST` | 非流式对话 |
| `/api/chat/stream` | `POST` | SSE 流式对话 |

## 3. 请求结构

### 3.1 非流式 / 流式统一请求体

```json
{
  "message": "帮我查一下武汉天气",
  "user_id": "u_001",
  "session_id": "s_001",
  "history": [
    {
      "role": "user",
      "content": "上一轮问题"
    },
    {
      "role": "assistant",
      "content": "上一轮回答"
    }
  ]
}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `message` | 是 | 当前用户输入 |
| `user_id` | 否 | 用户标识 |
| `session_id` | 否 | 会话标识，不传则服务端生成 |
| `history` | 否 | 历史消息 |

## 4. 非流式接口

### 4.1 `POST /api/chat`

#### 成功响应

```json
{
  "message": "武汉今天天气多云，最高温度 28 度。",
  "session_id": "s_001",
  "intent": "QUERY_WEATHER"
}
```

#### 返回要求

- 必须返回最终文本
- 必须返回 `session_id`
- 尽量返回识别到的 `intent`

## 5. 流式接口

### 5.1 `POST /api/chat/stream`

媒体类型：

```http
Content-Type: application/json
Accept: text/event-stream
```

### 5.2 事件协议

#### 事件 1: intent

```json
{
  "event": "intent",
  "data": {
    "intent": "QUERY_WEATHER"
  }
}
```

#### 事件 2: tool_start

```json
{
  "event": "tool_start",
  "data": {
    "tool": "query_weather"
  }
}
```

#### 事件 3: tool_result

```json
{
  "event": "tool_result",
  "data": {
    "tool": "query_weather",
    "result": {
      "city": "武汉",
      "weather": "多云"
    }
  }
}
```

#### 事件 4: response

```json
{
  "event": "response",
  "data": {
    "text": "武汉今天天气多云，适合出行。"
  }
}
```

#### 事件 5: done

```json
{
  "event": "done",
  "data": {
    "session_id": "s_001"
  }
}
```

#### 事件 6: error

```json
{
  "event": "error",
  "data": {
    "message": "weather tool timeout"
  }
}
```

## 6. 第一阶段支持的场景输入样例

### 6.1 TALK

输入：

```json
{
  "message": "你好，介绍一下你自己"
}
```

预期：

- 识别 `TALK`
- 直接返回自然语言回答

### 6.2 QUERY_WEATHER

输入：

```json
{
  "message": "帮我查一下武汉天气"
}
```

预期：

- 识别 `QUERY_WEATHER`
- 调用天气工具
- 返回天气结果

### 6.3 QUERY_SHIP

输入：

```json
{
  "message": "帮我查一下长江1号"
}
```

预期：

- 识别 `QUERY_SHIP`
- 解析船名
- 返回船舶基础信息

### 6.4 SAVE_ORDER

输入：

```json
{
  "message": "5月10日武汉装煤5000吨到南京"
}
```

预期：

- 识别 `SAVE_ORDER`
- 返回结构化运单预览

## 7. 第一阶段验收标准

### 7.1 启动验收

- 执行 `uvicorn backend.main:app --reload` 成功
- 打开 `/health` 返回 200

### 7.2 接口验收

- `/api/chat` 可正常返回 JSON
- `/api/chat/stream` 可正常返回 SSE
- 错误场景下也能返回标准错误事件

### 7.3 场景验收

至少满足以下 4 项：

| 场景 | 是否必须 |
|------|----------|
| TALK | 必须 |
| QUERY_WEATHER | 必须 |
| QUERY_SHIP | 二选一 |
| SAVE_ORDER | 二选一 |

也就是：

- `TALK` 必须通
- `QUERY_WEATHER` 必须通
- `QUERY_SHIP` 和 `SAVE_ORDER` 至少通一个

### 7.4 代码验收

- 有统一配置入口
- 有统一 LLM Client
- 有统一 Tool Registry
- LangGraph 主图可复用
- 至少具备基础测试
