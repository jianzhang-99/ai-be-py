# 第一阶段开发任务拆解

## 1. 任务目标

把第一阶段 MVP 方案拆成可执行的开发项，确保团队可以按顺序推进，而不是同时铺太多线。

## 2. 开发顺序

推荐顺序：

1. 先把服务跑起来
2. 再打通对话主链路
3. 再补意图识别
4. 再补工具调用
5. 最后补测试和文档

原因很简单：

- 先有运行中的骨架，调试路径短
- 再逐层加能力，容易定位问题

## 3. 任务分解

### A. 项目启动与配置

#### A1. 配置模块

- 新增 `backend/config.py`
- 定义 Settings 类
- 从 `.env` 读取：
  - 服务端口
  - LLM Key
  - 外部接口地址
  - debug 开关

验收标准：

- 本地缺省配置下可启动
- 必填配置缺失时提示清晰

#### A2. 路由整理

- 完成 `backend/api/routers/chat.py`
- 完成 `backend/api/routers/health.py`
- 在 `main.py` 中统一注册

验收标准：

- `/`
- `/health`
- `/api/chat`
- `/api/chat/stream`

都可访问

### B. 对话主链路

#### B1. ChatService 收口

- 统一 `chat_simple`
- 统一 `chat_stream`
- 自动补 `session_id`
- 将请求转换成 workflow 输入

验收标准：

- 同一请求路径不重复写编排逻辑
- 输入输出结构统一

#### B2. ChatWorkflow 落地

- 新建 `backend/graph/workflows/chat_workflow.py`
- 定义 LangGraph 主图
- 至少包含：
  - `intent_node`
  - `routing_node`
  - `tool_node`
  - `response_node`

验收标准：

- 能返回完整 state
- 流式/非流式都可以共用

#### B3. AgentState 落地

- 新建 `backend/graph/state/agent_state.py`
- 定义第一阶段最小状态字段

验收标准：

- 所有节点共享同一份状态约定
- 不出现节点之间随意追加脏字段

### C. 意图识别

#### C1. 规则识别

- 保留关键词规则
- 支持续接判断

第一阶段最低规则集：

- `天气 -> QUERY_WEATHER`
- `查船/船舶 -> QUERY_SHIP`
- `录单/运单 -> SAVE_ORDER`
- 其他默认 `TALK`

验收标准：

- 高频输入不依赖 LLM 即可识别

#### C2. LLM 识别

- 接入 `backend/infra/llm/client.py`
- 支持规则未命中时走模型分类

验收标准：

- LLM 返回值可解析
- 异常时自动回退到 `TALK`

### D. LLM 基础设施

#### D1. LLM Client

- 定义统一接口
- 默认实现 `DeepSeek`
- 支持：
  - 非流式 chat
  - 流式 stream

验收标准：

- 上层不直接依赖具体 SDK

#### D2. Prompt 组织

- 第一阶段先用代码内 prompt 或轻量模板
- 不急着做数据库版 prompt 管理

建议先做三套：

- 意图识别 prompt
- 通用回复 prompt
- 运单抽取 prompt

### E. Tool 体系

#### E1. Tool Registry

- 新建 `backend/tools/registry.py`
- 提供静态注册与查找

验收标准：

- 通过 `scene -> tool` 映射能找到执行器

#### E2. Weather Tool

- 新建 `backend/tools/weather.py`
- 输入：用户文本
- 输出：天气结构化结果 + 摘要文本

验收标准：

- 至少能返回一个稳定结果

#### E3. Ship Tool

- 新建 `backend/tools/ship.py`
- 输入：船名
- 输出：船舶基础信息

验收标准：

- 至少能返回 mock 或真实的单船结果

#### E4. Order Extract Tool

- 新建 `backend/tools/order.py`
- 输入：自然语言运单描述
- 输出：结构化字段

验收标准：

- 能抽出装货地、卸货地、日期、货物、吨位

### F. 节点实现

#### F1. routing_node

- 根据 intent 决定走 tool 还是 response

建议映射：

| Scene | 去向 |
|-------|------|
| TALK | response_node |
| QUERY_WEATHER | tool_node |
| QUERY_SHIP | tool_node |
| SAVE_ORDER | tool_node |

#### F2. tool_node

- 根据 scene 获取工具
- 执行工具
- 回填 `tool_result`

#### F3. response_node

- `TALK`：直接 LLM 生成
- 工具场景：把 `tool_result` 组装成最终返回文本

### G. 流式协议

#### G1. SSE 输出格式统一

建议每个事件都输出：

```json
{
  "event": "response",
  "data": {
    "text": "..."
  }
}
```

第一阶段至少支持：

- `intent`
- `tool_start`
- `tool_result`
- `response`
- `done`
- `error`

#### G2. 非流式聚合

- 非流式接口内部可以复用 workflow
- 将最终 `response` 聚合成一次性结果

### H. 测试

#### H1. 单元测试

- `IntentService`
- `ToolRegistry`
- `WeatherTool`
- `OrderExtractTool`

#### H2. 集成测试

- `/health`
- `/api/chat`
- `/api/chat/stream`

#### H3. Mock 策略

- 所有第三方接口在测试中必须可 Mock

## 4. 建议的里程碑

### M1. 服务可启动

完成内容：

- main
- router
- config
- health

结果：

- 本地服务可跑

### M2. 对话链路打通

完成内容：

- chat service
- workflow
- state
- response node

结果：

- `TALK` 可用

### M3. 第一个工具场景打通

完成内容：

- intent
- routing
- tool registry
- weather tool

结果：

- `QUERY_WEATHER` 可用

### M4. 航运能力上线

完成内容：

- ship tool
- order extract tool

结果：

- 至少一个航运业务场景稳定可演示

### M5. 质量收尾

完成内容：

- 单测
- 集成测试
- 文档补齐

结果：

- 第一阶段可以交付

## 5. 风险点

### 风险 1: LangGraph 先行导致调试复杂

建议：

- 第一阶段只做一条主图
- 不做过多条件边

### 风险 2: 外部服务不稳定

建议：

- 所有工具提供 mock fallback
- 先保证主链路不断

### 风险 3: 过早引入存储

建议：

- 第一阶段全部内存态
- 第二阶段再补 MySQL/Redis

### 风险 4: 事件协议反复改

建议：

- 第一阶段先把 SSE 事件定义稳定下来
- 后续只新增字段，不轻易改结构

## 6. 技术经理建议

第一阶段不要追求“像原 Java 版一样全”。

更合理的节奏是：

- 先做一个小而稳的 Python Agent 骨架
- 再把高价值业务能力一条条迁进去

如果第一阶段能把 `TALK + QUERY_WEATHER + QUERY_SHIP/SAVE_ORDER` 这条链跑稳，项目就已经迈过最危险的启动期了。
