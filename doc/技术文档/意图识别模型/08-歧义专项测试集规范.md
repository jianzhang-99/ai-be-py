# 歧义专项测试集规范（Ambiguity Test Set Specification）

## 1. 概述

歧义专项测试集（ambiguity_test.jsonl）是专门用于验证模型上下文继承能力和歧义消解能力的测试集，与常规测试集分开管理，长期跟踪模型在这类难例上的表现。

### 1.1 测试集定位

| 项目 | 说明 |
|------|------|
| 文件名 | ambiguity_test.jsonl |
| 数量 | 100~200 条 |
| 用途 | 专门测试上下文继承、歧义消解、澄清触发能力 |
| 管理方式 | 固定不动，长期跟踪，不参与训练 |
| 评测频率 | 每次模型迭代后必须评测 |

### 1.2 歧义类型分类

| 类型 | 说明 | 占比建议 |
|------|------|----------|
| 上下文继承 | 当前 query 依赖历史信息才能正确理解 | 40% |
| 船名地名歧义 | 同一个词既可以是船名也可以是地名 | 25% |
| 多意图冲突 | 一个 query 可能属于多个意图 | 20% |
| 信息不足澄清 | query 信息太少，无法确定意图 | 15% |

---

## 2. 上下文继承类（Context Inheritance）

### 2.1 定义

当前 query 的理解依赖于历史对话中的信息，用户省略了关键实体（如船名、航线等），需要从历史中继承。

### 2.2 标注要求

- `context_inherited: true` 必填
- `slots` 中需要标注从上下文继承的具体值
- `note` 中说明继承来源

### 2.3 样本示例

```json
{
  "id": "amb_0001",
  "type": "context_inheritance",
  "history": [
    {"role": "user", "content": "查船 俞垛79"},
    {"role": "assistant", "content": "已为您查到船舶俞垛79的位置"}
  ],
  "query": "俞垛在哪里",
  "label": {
    "intent": "QUERY_SHIP",
    "slots": {"ship_name": "俞垛79"},
    "need_clarify": false,
    "context_inherited": true,
    "note": "从上文 [USER] 查船 俞垛79 继承船名"
  },
  "expected": {
    "intent": "QUERY_SHIP",
    "context_inherited": true
  }
}
```

### 2.4 上下文继承场景清单

| 场景 | 历史 | 当前 query | 预期 slots |
|------|------|------------|------------|
| 查船后追问位置 | 查船 俞垛79 | 俞垛在哪里 | ship_name: 俞垛79 |
| 查船后追问轨迹 | 查船 俞垛79 | 这条船的轨迹 | ship_name: 俞垛79 |
| 找船后追问详情 | 南京港附近有没有船 | 有多少艘 | area_name: 南京港 |
| 查运价后追问具体 | 南京到南通砂石运价 | 5000吨呢 | route_from: 南京, route_to: 南通, cargo_weight: 5000吨 |
| 录单后追问状态 | 帮我录一条南京到重庆的运单 | 什么时候装 | route_from: 南京, route_to: 重庆 |

## 3. 船名地名歧义类（Ship vs Place Ambiguity）

### 3.1 定义

某些词既可能是船名的一部分，也可能是地名/区域名，模型需要结合上下文或触发澄清来消解歧义。

### 3.2 标注要求

- 无历史时：优先标注为地名，触发澄清
- 有历史（曾查过某船）时：继承为船名
- 记录歧义原因

### 3.3 样本示例

**无历史歧义**：

```json
{
  "id": "amb_0020",
  "type": "ship_place_ambiguity",
  "history": [],
  "query": "俞垛在哪里",
  "label": {
    "intent": "TALK",
    "slots": {},
    "need_clarify": true,
    "clarify_question": "您是在问俞垛这个地点在哪里，还是想查询俞垛79这条船的位置？"
  },
  "note": "无历史上下文，无法判断是地名还是船名，触发澄清",
  "expected": {
    "need_clarify": true
  }
}
```

**有历史继承为船名**：

```json
{
  "id": "amb_0021",
  "type": "ship_place_ambiguity",
  "history": [
    {"role": "user", "content": "俞垛79船在哪"},
    {"role": "assistant", "content": "俞垛79目前在长江下游"}
  ],
  "query": "俞垛还有多远到港",
  "label": {
    "intent": "QUERY_SHIP",
    "slots": {"ship_name": "俞垛79"},
    "need_clarify": false,
    "context_inherited": true
  },
  "note": "有历史上下文，明确是船名",
  "expected": {
    "intent": "QUERY_SHIP",
    "context_inherited": true
  }
}
```

### 3.4 常见歧义词清单

| 歧义词 | 可能含义 | 歧义场景 |
|--------|----------|----------|
| 俞垛 | 地名（俞垛镇）或船名（俞垛79） | "俞垛在哪里" |
| 南京 | 地名或船名（南京7/南京8） | "南京7在哪" |
| 长江 | 地名（长江）或船名（长江之星） | "长江之星" vs "长江下游" |
| 江北 | 地名（江北区域） | "江北有船吗" |
| 武穴 | 地名（武穴港）或船名 | "武穴在哪" |

## 4. 多意图冲突类（Multi-Intent Conflict）

### 4.1 定义

某些 query 可能同时符合多个 intent 的特征，需要结合上下文或业务规则来确定最终意图。

### 4.2 标注要求

- 记录所有可能的 intent
- 标注业务上最终确定的 intent 及原因
- 说明冲突原因

### 4.3 样本示例

**航线+吨位型（FIND_SHIP vs SAVE_ORDER）**：

```json
{
  "id": "amb_0040",
  "type": "multi_intent_conflict",
  "history": [],
  "query": "南京到南通5000吨砂石料明天装",
  "label": {
    "intent": "SAVE_ORDER",
    "slots": {
      "route_from": "南京",
      "route_to": "南通",
      "cargo_weight": "5000吨",
      "cargo_name": "砂石料",
      "date_time": "明天"
    },
    "need_clarify": false
  },
  "note": "产品确认为 SAVE_ORDER（发布运单），非 FIND_SHIP（找船）",
  "possible_intents": ["FIND_SHIP", "SAVE_ORDER"],
  "expected": {
    "intent": "SAVE_ORDER"
  }
}
```

**区域+货类型（FIND_SHIP vs QUERY_FREIGHT）**：

```json
{
  "id": "amb_0041",
  "type": "multi_intent_conflict",
  "history": [],
  "query": "武汉到南京煤炭多少钱",
  "label": {
    "intent": "QUERY_FREIGHT",
    "slots": {
      "route_from": "武汉",
      "route_to": "南京",
      "cargo_name": "煤炭"
    },
    "need_clarify": false
  },
  "note": "带'多少钱'明确是运价查询，不是找船",
  "possible_intents": ["QUERY_FREIGHT", "FIND_SHIP"],
  "expected": {
    "intent": "QUERY_FREIGHT"
  }
}
```

### 4.4 多意图冲突场景清单

| 场景 | query | 冲突 intent | 产品确认 |
|------|-------|-------------|----------|
| 航线+吨位 | 南京到南通5000吨砂石料明天装 | FIND_SHIP / SAVE_ORDER | SAVE_ORDER |
| 航线+货名+运价 | 武汉到南京煤炭多少钱 | QUERY_FREIGHT / FIND_SHIP | QUERY_FREIGHT |
| 港口+时间+船 | 镇江港明天有船吗 | FIND_SHIP / QUERY_WEATHER | FIND_SHIP |
| 船名+查询 | 华航118船到哪了 | QUERY_SHIP / DISPATCH_MONITOR | QUERY_SHIP |

## 5. 信息不足澄清类（Need Clarify）

### 5.1 定义

query 信息太少，无法确定用户真正意图，需要触发澄清。

### 5.2 标注要求

- `need_clarify: true` 必填
- `clarify_question` 必填，且要具体有针对性
- 不能用泛泛的"请问您想做什么"敷衍

### 5.3 样本示例

```json
{
  "id": "amb_0060",
  "type": "need_clarify",
  "history": [],
  "query": "帮我查一下",
  "label": {
    "intent": "TALK",
    "slots": {},
    "need_clarify": true,
    "clarify_question": "请问您想查船、找船、查运价，还是查天气？"
  },
  "expected": {
    "need_clarify": true
  }
}
```

```json
{
  "id": "amb_0061",
  "type": "need_clarify",
  "history": [
    {"role": "user", "content": "查船 俞垛79"},
    {"role": "assistant", "content": "已为您查到船舶俞垛79的位置"}
  ],
  "query": "还有吗",
  "label": {
    "intent": "TALK",
    "slots": {},
    "need_clarify": true,
    "clarify_question": "请问您还想查俞垛79的其他信息吗？比如预计到港时间或航行轨迹？"
  },
  "note": "有历史但'还有吗'太模糊，需要追问",
  "expected": {
    "need_clarify": true
  }
}
```

### 5.4 澄清触发场景清单

| 场景 | query | 正确澄清问题 |
|------|-------|--------------|
| 无目标查询 | 帮我查一下 | 请问您想查船、找船、查运价，还是查天气？ |
| 指代不明 | 还有吗 | 请具体说明您想查询什么 |
| 区域模糊 | 南京附近有船吗 | 请问您说的南京附近是指哪个港口？ |
| 船名歧义 | 南京7在哪里 | 您是在问南京7这条船，还是南京港附近？ |
| 时间模糊 | 明天能到吗 | 请问您说的是哪个运单/哪艘船？ |

## 6. 评估指标

### 6.1 上下文继承率（Context Inheritance Rate）

```
Context Inheritance Rate = 上下文继承正确数 / 总上下文继承测试数
```

### 6.2 歧义消解率

```
歧义消解率 = 歧义正确消解数 / 总歧义测试数
```

### 6.3 澄清拦截率

```
Clarify Recall = 该澄清的问题被正确拦截的比例 >= 80%
Clarify Precision = 标记澄清的问题中真正需要澄清的比例 >= 60%
```

## 7. 测试集管理

### 7.1 增补规则

当发现新的歧义 bad case 时：
1. 优先判断类型（上下文继承/歧义/多意图/澄清）
2. 标注后放入 ambiguity_test.jsonl
3. 从测试集中移除（不参与训练）
4. 下次评测时验证是否修复

### 7.2 固定不动原则

- ambiguity_test.jsonl 固定不动，只增不减
- 如果需要调整样本，需经过评审会议
- 用于长期跟踪模型在难例上的表现趋势
