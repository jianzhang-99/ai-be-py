# 意图识别模型技术方案

## 1. 技术目标

目标是构建一个适合航运场景的、**多轮可用的意图识别模型**，解决以下核心问题：

1. 意图识别不稳定，同一句话前后识别结果不一致
2. 航运专有名词（船名、地名、港口名）存在歧义
3. 多轮对话中上下文丢失，续问无法正确关联前文

---

## 2. 问题定义

结合实际业务场景，当前最典型的问题分为四类：

### 2.1 航运专名歧义

| 用户输入 | 实际含义 | 当前问题 |
|----------|----------|----------|
| `南京7` | 船名 | "南京"同时也是地名，容易触发地名理解 |
| `俞垛79` | 船名 | "俞垛"也是江苏泰州的地名 |
| `俞垛在哪里` | 查船位置 | 还是查地名？上下文不足时难以判断 |

### 2.2 意图与实体耦合

一句话往往同时包含意图和参数：

- `帮我查船：南京港附近` → 意图是 `FIND_SHIP`，参数是 `area_name=南京港附近`
- 只做纯 intent 分类不够，必须同时抽实体

### 2.3 多轮续问

```
上一轮：用户查了"俞垛79"的位置
下一轮：用户问"俞垛在哪里"
期望：自动关联到上一轮的"俞垛79"，继续查船位置
实际：可能被误解为查询"俞垛镇"的位置
```

### 2.4 线上路由稳定性

意图识别的输出直接决定下游链路走向，识别错误会比"拒绝回答"造成更大损失。导致

## 3. 方案结论

第一阶段推荐采用以下路线：

1. **基座选择**：使用 `Qwen` 7B~9B 量级指令模型
2. **训练方式**：`SFT` 领域微调，优先 `LoRA / QLoRA`
3. **训练目标**：不仅输出 `intent`，还要输出 `slots` 和 `need_clarify`
4. **RAG 定位**：暂不作为意图识别主方案，只为 `DOC_QA`、规则说明、帮助问答等场景预留

一句话总结：

> 先把"模型怎么判断"训好，再考虑"模型要不要查知识库"。

---

## 4. 为什么第一阶段先做微调，不先做 RAG

### 4.1 微调解决什么

- 模型是否理解航运行业的说法方式
- 模型是否能区分 `查船` 和 `区域找船`
- 模型是否能根据上下文继承上一轮主实体
- 模型是否能发现歧义并主动要求澄清

### 4.2 RAG 解决什么

- 模型需要查文档、查帮助中心、查业务规则

例如 `运吨吨怎么发布运单`、`某条业务规则是什么` 这类问题，才适合 RAG。

### 4.3 本项目判断

当前核心问题不是"缺知识"，而是**"不会判断"**。因此意图识别主线先做微调，文档问答、帮助问答后续再接 RAG。

---

## 5. 第一阶段模型目标

### 5.1 不做单标签分类

不推荐的目标形式：

```
输入一句话 → 输出 QUERY_SHIP / FIND_SHIP / TALK
```

这只解决了"打标签"问题，解决不了航运场景里的核心痛点。

### 5.2 推荐的目标形式

```
输入：当前问题 + 最近 1~3 轮上下文
输出：intent + slots + need_clarify
```

推荐输出格式：

```json
{
  "intent": "QUERY_SHIP",
  "slots": {
    "ship_name": "俞垛79",
    "area_name": null
  },
  "confidence": 0.91,
  "need_clarify": false,
  "clarify_question": null
}
```

歧义时输出示例：

```json
{
  "intent": null,
  "slots": {
    "ship_name": null,
    "area_name": "南京"
  },
  "confidence": 0.45,
  "need_clarify": true,
  "clarify_question": "你是想查船「南京7」的位置，还是想查询南京港附近的船舶？"
}
```

### 5.3 为什么要一起训练 slots

下游业务链路不仅需要知道"是什么意图"，还需要知道"参数是什么"：

- `QUERY_SHIP` 需要 `ship_name`
- `FIND_SHIP` 需要 `area_name`
- `SAVE_ORDER` 需要装货地、卸货地、货名、吨位等

如果只训 intent，后续还得补另一套抽取逻辑，整体更绕。

### 5.4 为什么要支持 need_clarify

线上系统最怕的是高置信误判。因此模型应支持在置信度不足或存在明显歧义时返回 `need_clarify = true`，主动触发澄清。

---

## 6. 模型输入输出设计

### 6.1 输入格式

统一为多轮指令格式：

```
你是航运助手的意图识别模型。
请根据历史对话识别当前问题的意图，并抽取关键槽位。
如果信息不足或存在明显歧义，请输出 need_clarify=true。

[历史对话]
用户：查船 俞垛79
助手：已为您查询到船舶俞垛79

[当前问题]
用户：俞垛在哪里
```

### 6.2 输出格式

统一输出 JSON：

```json
{
  "intent": "QUERY_SHIP",
  "slots": { "ship_name": "俞垛79", "area_name": null },
  "confidence": 0.91,
  "need_clarify": false,
  "clarify_question": null
}
```

## 7. 标签与槽位设计

### 7.1 意图集合（第一阶段）

| 意图 | 说明 |
|------|------|
| `QUERY_SHIP` | 查询单船信息（位置、详情） |
| `FIND_SHIP` | 区域找船（航线+船型+载重筛选） |
| `SAVE_ORDER` | 运单录入 |
| `QUERY_ORDER` | 运单查询 |
| `QUERY_WEATHER` | 天气查询 |
| `QUERY_FREIGHT` | 运价查询 |
| `QUERY_WATER_LEVEL` | 水位查询 |
| `DOC_QA` | 文档问答（后续接 RAG） |
| `TALK` | 闲聊 |
| `OTHER` | 其他 / 无法识别 |

### 7.2 槽位集合（第一阶段）

| 槽位 | 说明 |
|------|------|
| `ship_name` | 船名 |
| `ship_name_alias` | 船名别名/简称 |
| `area_name` | 区域名称 |
| `port_name` | 港口名称 |
| `route_from` | 起点 |
| `route_to` | 终点 |
| `cargo_name` | 货名 |
| `cargo_weight` | 货物重量 |
| `date_time` | 日期时间 |
| `order_field` | 运单字段 |

### 7.3 设计原则

- 一级意图先少，不要一开始拆太细
- 高频业务优先标注
- 槽位命名统一
- 歧义问题必须支持 `need_clarify`

## 8. 数据方案

### 8.1 样本结构

每条样本包含：

- `history`：历史对话（1~3 轮）
- `query`：当前问题
- `label.intent`：意图标签
- `label.slots`：槽位字典
- `label.need_clarify`：是否需澄清

示例：

```json
{
  "history": [
    {"role": "user", "content": "查船 俞垛79"},
    {"role": "assistant", "content": "已为您查询到船舶俞垛79"}
  ],
  "query": "俞垛在哪里",
  "label": {
    "intent": "QUERY_SHIP",
    "slots": { "ship_name": "俞垛79", "area_name": null },
    "need_clarify": false
  }
}
```

### 8.2 必须覆盖的样本类型

1. **单轮明确意图**：表达清晰，无歧义
2. **船名与地名歧义**：如"南京7"、"俞垛79"
3. **区域找船**：含区域、航线条件
4. **多轮续问**：省略主语、省略船名
5. **运单抽取**：含货物、地点、吨位信息
6. **极短追问**："呢？"、"呢在哪"
7. **模糊说法和口语化表达**：方言、缩写、不完整句

### 8.3 数据量建议

| 阶段 | 数量 | 说明 |
|------|------|------|
| 种子集 | 300~500 条 | 高质量，覆盖核心场景 |
| 扩展训练集 | 1000~3000 条 | 覆盖更多变体 |
| 歧义专项测试集 | 100~300 条 | 专门测试歧义处理能力 |

对第一阶段来说，**数据质量比规模更重要**。

---

## 9. 训练方案

### 9.1 基座模型建议

- 选择 `Qwen` 指令模型，7B~9B 量级
- 重点关注：中文理解能力、结构化输出能力、训练成本

### 9.2 训练方式建议

- **方式**：SFT
- **方法**：优先 LoRA / QLoRA，不做全参数微调
- **原因**：显存成本低、上手简单、训练和回滚都更轻

### 9.3 为什么不建议第一阶段做全参数微调

第一阶段最重要的是验证模型在航运场景的效果，全参数微调成本高、周期长，不适合快速迭代。

---

## 10. 推理与系统接入方案

### 10.1 接入方式

模型作为独立的意图识别服务，接入现有编排层：

```text
ChatWorkflow
  → intent_node
      → IntentModelService（调用意图识别模型）
      → 返回 intent + slots + confidence + need_clarify
  → routing_node（根据 intent 路由）
  → tool_node / response_node
```

### 10.2 这样做的好处

- 不推翻现有工程结构
- 可与旧规则并行运行
- 容易做灰度和 A/B 对比
- 出问题时方便回退

### 10.3 路由逻辑

| 模型输出 | 后续动作 |
|----------|----------|
| `need_clarify = true` | 直接返回澄清问题 |
| `intent = QUERY_SHIP` | 走单船查询链路 |
| `intent = FIND_SHIP` | 走区域找船链路 |
| `intent = SAVE_ORDER` | 走运单抽取链路 |
| `intent = DOC_QA` | 后续可接 RAG |

---

## 11. RAG 在本方案中的位置

### 11.1 适合接 RAG 的场景

- `DOC_QA`：文档问答
- 帮助中心问答
- 业务规则解释
- 接口或流程说明

### 11.2 不适合 RAG 负责的部分

- 主意图识别
- 主实体歧义判断
- 多轮续问主判定

这些是"理解问题"，不是"查资料问题"。

### 11.3 推荐组合方式

```text
用户输入
  → 意图识别模型
      → 如果是 DOC_QA → 走 RAG
      → 如果是 QUERY_SHIP / FIND_SHIP / SAVE_ORDER → 走业务工具
```

---

## 12. 风险点与缓解方式

| 风险 | 表现 | 缓解方式 |
|------|------|----------|
| 数据太少 | 泛化差，覆盖不了长尾 | 先做高质量种子集 |
| 标注不一致 | 同类问题学不稳 | 先写清标注规范 |
| JSON 输出不稳 | 推理结果难解析 | 统一模板并加强训练约束 |
| 歧义误判 | 船名/地名混淆 | 建立专项测试集 |
| 上下文失效 | 多轮理解差 | 样本必须带历史对话 |

### 12.1 第一阶段最容易犯的错

1. 一上来就想把微调 + RAG + 复杂 Agent 全部一起做
2. 只标 intent，不标 slots
3. 训练样本不带上下文
4. 不给歧义问题设置 `need_clarify`

---

## 13. 第一阶段完成定义

当以下条件全部满足时，第一阶段视为完成：

1. ✅ 已完成意图与槽位体系定义
2. ✅ 已完成一版训练样本整理（≥300 条高质量样本）
3. ✅ 已完成至少一次 SFT 训练（LoRA）
4. ✅ 模型能稳定输出合法 JSON
5. ✅ 核心样本上优于当前规则方案
6. ✅ 能接入现有 `intent_node` 本地联调
7. ✅ `DOC_QA` 仍保留为后续 RAG 扩展位

---

## 14. 最终结论

第一阶段最合适的路线不是"微调 + RAG 全套"，而是：

1. 先训练一个适合航运场景的 SFT 意图识别模型
2. 同时把 `slots` 和 `need_clarify` 一起纳入训练
3. 先把"查船 / 找船 / 运单 / 续问 / 歧义"做稳
4. 后续再把 `DOC_QA` 等场景接到 RAG

> 先把模型训成一个会判断、会抽参数、会发现歧义的航运意图识别器，<br>再考虑给它接知识库。
