# 意图识别模型调研报告（第一阶段）

## 1. 调研目标

本报告基于“训练一个意图识别模型来做意图识别”这一前提展开，不再讨论是否要训练，而是重点回答以下问题：

1. 截至 2026 年 5 月 11 日，市面上主流的意图识别实现方案是什么样的
2. 当前更前沿的实现趋势是什么样的
3. 结合本项目的航运场景，第一阶段应如何选型和落地

本报告聚焦的是任务型对话中的 NLU/SLU 能力，包括：

- 意图识别（Intent Classification）
- 槽位抽取（Slot Filling / Entity Extraction）
- 多轮上下文理解（Multi-turn Intent Classification）
- 歧义消解与路由（Orchestration / Guard）

## 2. 结合项目现状的目标问题定义

结合现有文档和代码，项目当前不是简单的“查船/天气/录单”单轮分类问题，而是一个典型的领域化多轮意图识别问题，核心难点有四类：

1. 航运专名歧义
   - `南京7` 可能是船名，也可能触发地名理解
   - `俞垛79` 是船名，但 `俞垛` 也可能被理解为地名

2. 意图与实体耦合
   - `帮我查船：南京港附近` 既有意图，也有区域槽位
   - 只做纯 intent label 分类不够，必须同时抽实体

3. 多轮续问
   - 上一轮查了 `俞垛79`
   - 下一轮 `俞垛在哪里` 本质是上下文续接

4. 线上路由稳定性
   - 识别结果不仅要“像对了”
   - 还要能够稳定路由到单船查询、区域找船、运单录入等下游链路

因此，这个项目真正要训练的不是一个“只输出一个类别”的文本分类器，而更接近一个领域化语义解析模型：

```text
输入：当前用户问题 + 最近若干轮上下文
输出：intent + slots/entities + confidence + 是否需澄清
```

## 3. 当前市面上主流实现路线

从主流产品和工程实践看，意图识别已经不再是单一模型问题，而是“意图 + 实体 + 对话控制”的组合系统。

### 3.1 主流路线一：Intent + Entity 联合建模

这是目前最经典、也最广泛落地的路线。

典型做法：

- 用一个编码器理解整句语义
- 句级输出 intent
- token 级输出 entity/slot
- 两个任务共享底层表示并联合训练

为什么这是主流：

- 意图和实体天然互相约束
- `查船` 和 `船名/区域` 必须一起看才更稳定
- 对业务系统来说，下游真正需要的是“意图 + 参数”

微软在 CLU 最佳实践里明确强调，schema 设计应把“动作/查询”建模为 intents，把“完成动作所需的信息”建模为 entities，而不是把所有概念都塞进 intents 中。  
这和你项目高度一致：`QUERY_SHIP` 是 intent，`ship_name`、`area_name` 是 entity/slot。

对你项目的映射：

- `QUERY_SHIP` / `FIND_SHIP` / `SAVE_ORDER` 是 intents
- `ship_name` / `port_name` / `area_name` / `cargo_name` / `tonnage` 是 slots

### 3.2 主流路线二：规则/词典/实体库与模型混合

工业界很少只靠纯神经网络直接上线，通常都会混合以下能力：

- 关键词和模式规则
- 领域词典、实体别名表
- 预置实体类型
- 模型预测后的阈值控制和 fallback

Azure CLU 的实体能力就是“learned component + list component + prebuilt component”的组合思路，本质上就是：

- 让模型学上下文
- 让词典补领域实体
- 让预置实体补通用类型
- 再通过 required component 约束误识别

这类设计非常适合你的航运场景，因为：

- 船名有明显领域词汇和命名模式
- 港口、锚地、河段、船型都适合词典化
- 模型负责泛化，词典负责兜底

### 3.3 主流路线三：多轮路由或编排层

对于复杂业务，主流系统不只训练一个 intent classifier，而会增加“上层编排器”。

微软的 orchestration workflow 典型地把多个 CLU/CQA 能力接到统一入口，让系统先判断该把请求路由到哪个目标能力。

Rasa 的 CALM 也体现了类似思路：LLM 负责理解用户表达，但对话执行仍然沿着预定义流程和 memory/slots 走，强调“结构化理解 + 可控执行”。

这说明行业共识已经比较明确：

- 单轮 intent classifier 仍然重要
- 但它通常只是整个对话系统的一层
- 真正上线时，还需要 memory、slot、guard、orchestration

对你项目来说，训练模型的同时，最终仍要接入：

- 单船查询链路
- 区域找船链路
- 运单录入链路
- 泛问答/闲聊兜底链路

### 3.4 主流路线四：阈值 + no-match + 澄清

Google Dialogflow CX 明确使用 intent confidence 和 classification threshold：

- 高于阈值则命中 intent
- 低于阈值则进入 no-match

这类机制在生产上非常重要，因为线上最怕的不是“不识别”，而是“高置信误识别”。

对你的场景尤其关键：

- `南京7` 低置信时应进入澄清
- `俞垛在哪里` 在船名/地名冲突时应进入澄清

也就是说，主流系统并不追求“每条都直接判死”，而是允许：

- 返回 `None`
- 返回候选
- 触发澄清问题

## 4. 当前前沿实现趋势

如果说主流路线是“编码器联合建模 + 规则混合”，那么 2024 到 2025 年更前沿的趋势主要有四条。

### 4.1 前沿趋势一：多轮上下文建模成为核心

过去很多 intent 模型默认是单轮文本分类，但现在多轮意图识别已成为重点。

LARA（2024，EMNLP）明确把问题定义为 multi-turn intent classification，并采用：

- 小模型微调
- 检索增强
- 与 LLM 结合的上下文利用

其核心思想非常适合你的项目：模型不能只看当前这句话，还要看历史对话、历史意图和相似案例。

这对你的场景几乎是刚需：

- `俞垛在哪里`
- `南京呢`
- `附近还有吗`

这类输入脱离上下文几乎不可判，必须做 multi-turn intent modeling。

### 4.2 前沿趋势二：意图识别与槽位抽取进一步深度联合

MMCL（2024）继续强化一个方向：不是简单地把 intent 和 slot 放在一起训练，而是通过多层、多粒度对比学习，让句子级 intent 与 token 级 slot 相互指导。

这代表当前研究界的趋势是：

- 不把 intent 和 slot 看成两个松耦合任务
- 而是把它们视为同一个 semantic frame 的不同视角

对你的业务意义很直接：

- 如果模型识别到 `ship_name=俞垛79`
- 那么 intent 更可能是 `QUERY_SHIP`
- 如果模型识别到 `area_name=南京港附近`
- 那么 intent 更可能是 `FIND_SHIP`

也就是说，前沿方向正在不断加强“intent 与实体的互相约束”。

### 4.3 前沿趋势三：指令微调 LLM 做结构化 IC+SF

ILLUMINER（2024，LREC-COLING）展示了一条非常重要的路线：

- 把 intent classification 和 slot filling 统一成生成任务
- 用 instruction-tuned LLM 做 few-shot 或 PEFT 微调
- 输出结构化结果

这类路线的价值在于：

- 对低样本更友好
- 易于迁移到新领域
- 对复杂表达和长尾问法泛化更强

对你这个项目，这条路线意味着可以考虑：

- 用中等规模指令模型做 LoRA/PEFT 微调
- 输出 JSON 结构：`intent + slots + confidence + need_clarify`

这已经不再是传统分类器，而是领域化语义解析器。

### 4.4 前沿趋势四：Entity-level 与 Chain-of-Intent

ECLM（ACL 2025）进一步提出：

- 把 slot filling 重构为 entity recognition
- 用 Chain of Intent 做 step-by-step multi-intent recognition

它背后的方向很值得注意：

- 面对复杂、歧义、多意图输入时
- 先明确实体，再逐步推断意图链

这和你航运场景高度贴合，因为你的很多误判，本质不是分类器不会分类，而是实体识别和推断顺序不对。

例如：

- `帮我查船：南京港附近`
  - 先识别 `南京港附近` 是区域
  - 再判断这是区域找船而不是单船查询

- `俞垛在哪里`
  - 先结合历史判断 `俞垛` 是否指向上一轮的 `俞垛79`
  - 再决定是否走单船位置查询

## 5. 主流方案对比

### 5.1 传统文本分类模型

形态：

- BERT/RoBERTa 类编码器
- 输入单句
- 输出单个 intent label

优点：

- 训练简单
- 推理快
- 易于上线

缺点：

- 无法天然解决实体抽取
- 对多轮上下文能力弱
- 对领域歧义适应差

结论：

只能做 very baseline，不适合作为你项目的目标方案。

### 5.2 联合 IC+SF 模型

形态：

- 一个共享编码器
- intent head + slot head
- 常见是 softmax + CRF / span extractor

优点：

- 最符合任务型对话场景
- 推理成本低于 LLM
- 训练与评估成熟

缺点：

- 多轮能力仍需额外设计
- 长尾泛化不如指令模型

结论：

这是最稳妥的第一阶段主模型路线。

### 5.3 检索增强多轮模型

形态：

- 当前轮 + 历史轮
- 相似样本召回
- 编码器或小模型分类

优点：

- 对上下文和长尾问法更稳
- 对项目里的歧义场景帮助很大

缺点：

- 系统复杂度更高
- 需要维护样本库和检索链路

结论：

这是第一阶段值得预留、第二阶段值得增强的路线。

### 5.4 指令模型 LoRA/PEFT 方案

形态：

- 中小型 instruction-tuned LLM
- 以 JSON 结构化输出 intent 和 slots
- 用 PEFT/LoRA 做领域微调

优点：

- 对复杂表达泛化更强
- 可自然利用多轮上下文
- 可统一 intent、slot、clarify 判断

缺点：

- 成本和延迟高于 encoder
- 线上可控性和稳定性需要更强评估

结论：

适合作为“前沿主线”或第二条试验线，但不建议作为第一阶段唯一生产方案。

## 6. 结合本项目的推荐调研结论

如果站在“就是要训练一个意图识别模型”的前提下，结合行业主流与前沿，我的结论是：

### 6.1 第一阶段最合适的目标不是“单分类模型”

你应该定义为：

```text
多轮领域意图识别 + 槽位抽取 联合模型
```

最低输出结构建议为：

```json
{
  "intent": "QUERY_SHIP",
  "slots": {
    "ship_name": "俞垛79",
    "area_name": null
  },
  "confidence": 0.91,
  "need_clarify": false
}
```

### 6.2 第一阶段推荐采用“双轨调研结论”

主线方案：

- 训练一个 encoder-based 联合 IC+SF 模型
- 输入为 `当前句 + 截断上下文`
- 输出 `intent + slots`

前沿备选方案：

- 训练一个 instruction-tuned LLM 的 LoRA 版本
- 让模型直接生成结构化语义帧

这样做的原因：

- 主线方案足够稳、够快、可评估
- 备选方案可以验证前沿能力和未来上限

### 6.3 第一阶段不建议直接做“纯 LLM 在线判意图”

虽然现在很多系统会接 LLM，但从行业实现看，真正主流稳定的方案不是“让 LLM 自由发挥判意图”，而是：

- 用 schema 约束
- 用 slot/entity 约束
- 用阈值和 guard 约束
- 用路由层控制执行

因此，即便训练 LLM，也建议做成“结构化预测模型”，而不是“开放式聊天模型”。

## 7. 面向本项目的训练任务定义建议

### 7.1 推荐标签体系

建议第一阶段先收敛为较稳定的一级意图：

- `QUERY_SHIP`
- `FIND_SHIP`
- `SAVE_ORDER`
- `QUERY_ORDER`
- `QUERY_WEATHER`
- `QUERY_FREIGHT`
- `QUERY_WATER_LEVEL`
- `DOC_QA`
- `TALK`
- `OTHER` 或 `NONE`

### 7.2 推荐槽位体系

结合当前业务，建议优先建设这些 slots：

- `ship_name`
- `ship_name_alias`
- `area_name`
- `port_name`
- `route_from`
- `route_to`
- `cargo_name`
- `cargo_weight`
- `date_time`
- `order_field`

### 7.3 多轮输入格式建议

第一阶段训练数据建议不要只保留当前句，建议统一成：

```text
[HIS_1] 用户：查船 俞垛79
[HIS_1] 助手：已为您查询到船舶俞垛79
[CUR] 用户：俞垛在哪里
```

然后输出：

```json
{
  "intent": "QUERY_SHIP",
  "slots": {
    "ship_name": "俞垛79"
  }
}
```

如果不做这种多轮格式，模型很难真正学会你的核心问题。

## 8. 第一阶段建议调研后的技术路线

### 8.1 路线 A：主生产方案

联合 IC+SF 模型，推荐结构：

- 中文预训练编码器
- 上下文拼接输入
- sentence-level intent head
- token-level slot head
- 可选 CRF/span decoder

训练目标：

- intent cross entropy
- slot tagging loss
- 可加权联合损失

适用原因：

- 工程成熟
- 训练成本可控
- 适合你当前阶段快速形成评估基线

### 8.2 路线 B：前沿探索方案

结构化生成式模型，推荐形态：

- 中小型指令模型
- 统一生成 intent + slots + clarify
- 使用 LoRA/PEFT 微调

适用原因：

- 对长尾问法和复杂歧义更有潜力
- 未来可以自然扩展到更复杂对话理解

### 8.3 路线 C：增强层

无论 A 还是 B，都建议配套：

- 航运实体词典
- 船名/港口别名表
- 相似样本检索库
- 置信度阈值
- clarify 模块

这不是“额外功能”，而是生产可用性的必要组成。

如果按调研后的执行视角，第一阶段建议产出不是代码，而是以下资产：

4. 基线方案说明

   - Intent Accuracy

   - Clarification Recall

## 10. 最终结论

截至 2026 年 5 月 11 日，主流和前沿实现可以概括为一句话：

```text
主流是“Intent + Entity 联合建模 + 规则/词典混合 + 阈值路由”，
前沿是“多轮上下文 + 检索增强 + 指令微调 LLM + 结构化语义帧输出”。
```

结合你这个项目，如果第一阶段就是要训练意图识别模型，我的明确建议是：

1. 不要把目标定义成“单句 intent 分类器”
2. 要把目标定义成“多轮 IC+SF 联合模型”
3. 第一阶段主线采用 encoder-based 联合模型
4. 同时开一条 LoRA 结构化生成试验线
5. 必须把航运实体词典、歧义样本、澄清机制一起纳入方案

如果只给一句最落地的话：

先训练一个“多轮上下文 + 意图分类 + 槽位抽取”的联合模型，把 `船名/区域/续问` 三件事一起学进去，这才是最符合你项目实际问题的第一阶段模型方向。
