# BERT 意图识别模型评估指标设计

## 1. 背景与现状

### 1.1 项目背景

AI-dundun 是航运助手智能客服，当前意图识别依赖规则匹配 + LLM 兜底的两层架构，存在以下痛点：

| 痛点 | 表现 | 示例 |
|------|------|------|
| 上下文意图混乱 | 用户省略关键信息时无法继承上下文 | "俞垛在哪里" → 应继承上文"俞垛79"查船，而非问地名 |
| 专用名词歧义 | 船名与地名/区域名难以区分 | "南京7"是船名，"南京港"是区域 |
| 弱意图表达难识别 | 无明显动词的业务话术 | "南京到南通5000吨砂石料明天装" 无动词但应为找船/录单 |
| LLM 成本与稳定性 | 依赖大模型兜底，响应慢、成本高 | 简单问题也走 LLM，浪费资源 |

### 1.2 当前方案架构

```
用户输入 → 规则优先判断 → BERT 模型 → LLM 兜底/澄清 → 下游工具
```

| 层级 | 现状 | 问题 |
|------|------|------|
| 规则层 | 关键词强关联 | 覆盖简单场景，但无法处理歧义和上下文 |
| 模型层 | 语义相似度向量库 | 依赖人工标注样本，泛化能力弱 |
| LLM 层 | 大模型兜底 + 二次确认 | 成本高、响应慢、稳定性差 |

### 1.3 本阶段目标

训练轻量级 BERT 意图识别模型，解决上述痛点，实现：
- 上下文意图继承
- 船名/地名歧义消解
- 弱意图表达识别
- 降低 LLM 调用比例

## 2. 评估体系设计

### 2.1 三层评估架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        评估体系三层架构                          │
├─────────────────────────────────────────────────────────────────┤
│  技术评估（模型效果）  │  系统评估（服务性能）  │  业务评估（价值指标） │
│  ─ 意图 准确率       │  ─ 推理延迟 P99         │  ─ BERT 采纳率        │
│  ─ 槽位 F1             │  ─ QPS                  │  ─ LLM 兜底率         │
│  ─ 澄清 拦截率      │  ─ 可用性               │  ─ Bad Case 率       │
│  ─ 完全命中率         │  ─ 资源消耗             │  ─ 问题解决率         │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 评估数据集设计

#### 数据集划分

| 数据集 | 用途 | 数量要求 | 说明 |
|--------|------|----------|------|
| 训练集 | 模型训练 | 1500~3000 条 | 覆盖全部 14 个意图，P0 场景 200+ 条/类 |
| 验证集 | 调参/早停 | 300~500 条 | 每个意图都要有样本 |
| 测试集 | 最终评测 | 400~800 条 | 固定不动，长期跟踪 |
| 歧义专项集 | 上下文/歧义测试 | 100~200 条 | 专门验证上下文继承和澄清能力 |
| Bad Case 回流集 | 回归测试 | 持续积累 | 线上 bad case 优先入测试集 |

#### 数据质量要求

1. **标签一致性**：每条样本需经过双人标注，Kappa 系数 >= 0.85
2. **分布合理性**：每类意图样本量相差不超过 5 倍
3. **样本多样性**：同意图样本需覆盖多种表达方式，避免模板重复

## 3. 技术评估指标（模型效果）

### 3.1 意图识别评估指标

| 指标 | 定义 | 计算方式 | 行业水平 | 目标值 | 可行性 |
|------|------|----------|----------|--------|--------|
| **Intent Accuracy** | 所有预测中意图正确的比例 | correct / total | >90% | **>= 88%** | ✅可达 |
| **Intent Macro F1** | 所有意图类别 F1 的算术平均 | mean(F1_i) | >85% | **>= 82%** | ✅可达 |
| **Intent Weighted F1** | 按样本量加权的 F1 | sum(n_i * F1_i) / sum(n_i) | >87% | **>= 85%** | ✅可达 |

**为什么要用 Macro F1 而不是单纯的 Accuracy？**

意图类别不平衡（如 TALK 闲聊占比高），Accuracy 会被高频类主导。Macro F1 对所有类一视同仁，能发现"把冷门意图都误判成 TALK"的问题。

### 3.2 槽位抽取评估指标

| 指标 | 定义 | 计算方式 | 目标值 | 可行性 |
|------|------|----------|--------|--------|
| **Slot Entity F1** | 按完整实体评估，非 token 级 | F1(entities_pred, entities_true) | **>= 78%** | ✅可达 |
| **Slot Token F1** | 按 token 评估（辅助参考） | F1(tokens_pred, tokens_true) | **>= 85%** | ✅可达 |
| **Slot Type Accuracy** | 槽位类型命中（位置对且类型对） | correct_type / total | **>= 80%** | ✅可达 |

**Entity F1 计算示例**

```
输入：南京 到 南通 5000吨 砂石料 明天 装
标注实体：
  - route_from: 南京
  - route_to: 南通
  - cargo_weight: 5000吨
  - cargo_name: 砂石料
  - date_time: 明天

预测实体：
  - route_from: 南京
  - route_to: 南通
  - cargo_weight: 5000吨
  - cargo_name: 砂石 (漏了"料")
  - date_time: 明天

Entity F1 = 4/5 (预测4个完整实体，1个不完整)
```

### 3.3 澄清判断指标

| 指标 | 定义 | 目标值 | 可行性 |
|------|------|--------|--------|
| **Clarify Recall** | 该澄清的问题被正确拦截的比例 | **>= 80%** | ✅可达 |
| **Clarify Precision** | 标记澄清的问题中真正需要澄清的比例 | **>= 60%** | ✅可达 |
| **Clarify F1** | Clarify 的综合 F1 | **>= 68%** | ✅可达 |

**阈值设定说明**

- Clarify Recall 设置 >= 80% 是因为第一阶段允许少量漏过，但关键是不能让系统瞎猜错误答案
- Clarify Precision 设置 >= 60% 是因为当前基准较低（依赖规则），提升需要时间
- 后续可根据业务反馈调整阈值

### 3.4 端到端指标

| 指标 | 定义 | 目标值 | 说明 |
|------|------|--------|------|
| **Exact Match** | intent + slots + clarify 全部正确的比例 | **>= 60%** | 业务可直接使用的完整结果 |
| **Intent + Slots Match** | intent 正确且 slots 实体完整匹配 | **>= 65%** | 不要求 clarify 完全匹配 |
| **Context Inheritance Rate** | 上下文继承测试集中继承正确的比例 | **>= 85%** | 专门衡量上下文能力 |

## 4. 系统评估指标（性能指标）

### 4.1 性能指标

| 指标 | 定义 | 测量方式 | 目标值 | 说明 |
|------|------|----------|--------|------|
| **推理延迟 P50** | 50% 请求的推理时间 | 服务端埋点 | < 30ms | 单次 BERT 前向传播 |
| **推理延迟 P99** | 99% 请求的推理时间 | 服务端埋点 | < 80ms | 含序列化开销 |
| **端到端延迟 P99** | 用户请求到返回的完整时间 | API 日志 | < 200ms | 含规则 -> BERT -> 组装 |
| **首 Token 延迟** | 流式输出首字时间 | SSE 日志 | < 500ms | 不适用于本场景（同步返回） |
| **吞吐量** | 每秒处理请求数 | 压测工具 | > 50 QPS | 单机 1 实例 |

### 4.2 可用性指标

| 指标 | 定义 | 目标值 | 说明 |
|------|------|--------|------|
| **服务可用性** | 服务正常运行时间占比 | 监控 | >= 99.5% | 含模型加载时间 |
| **模型加载时间** | 服务冷启动到可用的时间 | 启动日志 | < 30s | GPU 机型 |
| **失败率** | 请求失败（超时/异常）的比例 | 监控 | < 0.5% | |
| **自动恢复时间** | 模型服务异常后自动恢复时间 | 告警记录 | < 60s | |

### 4.3 资源消耗指标

| 指标 | 定义 | 目标值 | 说明 |
|------|------|--------|------|
| **GPU 显存占用** | 模型推理时显存峰值 | nvidia-smi | < 4GB | batch_size=16, seq_len=256 |
| **CPU 内存占用** | 服务进程内存 | 监控 | < 8GB | 含模型权重 |
| **模型文件大小** | pytorch_model.bin 大小 | 文件大小 | < 500MB | bert-base-chinese 微调后 |

## 12. 附录

### 12.1 指标计算公式

**Macro F1**
```
Macro F1 = (1/K) * Σ F1_k
其中 F1_k = 2 * Precision_k * Recall_k / (Precision_k + Recall_k)
K = 意图类别数量
```

**Entity F1（槽位实体级）**
```
Entity F1 = 2 * Entity_Precision * Entity_Recall / (Entity_Precision + Entity_Recall)
Entity_Precision = 预测正确的完整实体数 / 预测的实体总数
Entity_Recall = 预测正确的完整实体数 / 标注的实体总数
```

**Clarify F1**
```
Clarify_Precision = TP_clarify / (TP_clarify + FP_clarify)
Clarify_Recall = TP_clarify / (TP_clarify + FN_clarify)
Clarify_F1 = 2 * P * R / (P + R)
```
