# 意图识别模型调研报告

## 1. 调研目标

了解当前主流和前沿的意图识别方案，结合航运场景给出第一阶段落地建议。

---

## 2. 航运场景的核心难点

### 2.1 船名和地名分不清

- `南京7` 是船名，但"南京"也是地名
- `俞垛79` 是船名，"俞垛"也是江苏泰州的地名
- 光看单句话无法判断，必须结合上下文

### 2.2 意图和参数要一起抽

- `帮我查船：南京港附近` → 意图是"区域找船"，参数是"南京港附近"
- 只判断意图不够，必须同时抽参数

### 2.3 多轮对话会丢失上下文

- 上一轮查了"俞垛79"
- 下一轮问"俞垛在哪里" → 本质是在问上一轮的船，不是问地名

**结论**：不能只做单句分类器，要做"多轮 + 意图 + 参数 + 歧义处理"联合模型。

---

## 3. 四个核心能力

| 能力 | 通俗理解 | 例子 |
|------|----------|------|
| **意图识别** | 判断用户想干什么 | "今天武汉天气" → 查天气 |
| **槽位抽取** | 提取关键参数 | "武汉到南京" → 出发地、目的地 |
| **多轮理解** | 结合上文理解省略表达 | 上文查船"俞垛79"，下文问"在哪" |
| **歧义消解** | 搞不清就反问用户 | "南京7" → 问清楚是船还是地名 |

---

## 4. 主流方案对比

| 方案 | 适合做什么 | 缺点 |
|------|-----------|------|
| **传统分类器**（BERT） | 单句分类 | 抽不了参数、不懂上下文 |
| **联合模型**（intent + slot 一起训） | 意图+参数一起抽、最成熟 | 多轮上下文需要额外设计 |
| **检索增强**（当前轮+历史轮） | 多轮理解好 | 系统复杂、要维护样本库 |
| **LLM 微调**（Qwen LoRA） | 泛化强、能直接生成结构化结果 | 成本高、延迟大 |

---

## 5. 第一阶段推荐路线

### 主线：联合模型（encoder-based）

- 中文预训练编码器
- 同时输出 intent + slots
- 加上下文输入（历史对话）

### 备选：LLM 微调（Qwen LoRA）

- 用指令模型做生成式输出
- 成本高，作为探索线

### 必加：航运词典兜底

- 船名别名表、港口名、区域名
- 模型认不出来的，词典来补充

---

## 6. 最终输出格式

训练模型统一输出这个结构：

```json
{
  "intent": "QUERY_SHIP",
  "slots": { "ship_name": "俞垛79", "area_name": null },
  "confidence": 0.91,
  "need_clarify": false,
  "clarify_question": null
}
```

歧义时返回：

```json
{
  "intent": null,
  "slots": { "ship_name": null, "area_name": "南京" },
  "confidence": 0.45,
  "need_clarify": true,
  "clarify_question": "你想查船「南京7」的位置，还是想查询南京港附近的船舶？"
}
```
